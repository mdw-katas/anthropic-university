# CS 350 — Software Engineering

**Prerequisites:** CS 201
**Language:** team/project choice

## Course Description

Building software that survives contact with time, teams, and change:
requirements, design, testing, delivery, and maintenance. For a 16-year
practitioner this course is less about new information and more about
examining professional practice critically — connecting what you do to the
evidence and principles behind it, and filling gaps that habit papers over.

## Learning Objectives

- Elicit and specify requirements; distinguish essential from accidental
  complexity.
- Apply design principles (coupling/cohesion, SOLID, dependency direction)
  and architectural patterns deliberately, with their costs in view.
- Practice test-driven development rigorously and articulate the design
  pressure tests exert.
- Explain delivery engineering: version control workflows, CI/CD, code
  review, feature flags, observability.
- Evaluate process (agile and its ancestors) as an engineering tradeoff,
  not a ritual.

## Topic Outline

1. What makes software engineering hard: essential vs. accidental
   complexity; a brief history of the field.
2. Requirements: user stories, specifications, and the cost of ambiguity.
3. Design principles: information hiding, coupling and cohesion, SOLID
   examined critically.
4. Architecture: layering, hexagonal/ports-and-adapters, monolith vs.
   services; documenting decisions (ADRs).
5. Testing in depth: unit/integration/system, TDD (red-green-refactor),
   property-based testing, test doubles, coverage honesty.
6. Refactoring: identifying smells, making change safe.
7. Code review, version control workflows, and trunk-based vs. branch-heavy
   development.
8. Continuous integration and delivery; release engineering; feature flags.
9. Operating software: observability, incident response, postmortems.
10. Estimation, technical debt, and process: what the evidence actually
    says; ethics and professional responsibility.
