Contracts — Blame Is a Design Tool

You have spent sixteen years writing docstrings, API docs, and code review comments that say things like "callers must not pass nil here." This lesson gives that instinct its formal name — the contract — and its three clauses: preconditions, postconditions, and invariants. The payoff is not vocabulary. It is that a precise contract tells you, before any debugger opens, whose bug it is.

1. The three clauses

def withdraw(balance, amount):
    """Return the balance after withdrawing amount.

    Precondition:  0 <= amount <= balance
    Postcondition: result == balance - amount, and result >= 0
    """
    return balance - amount

A precondition is what must be true at the moment of the call — the caller's obligation. A postcondition is what the function guarantees on return, provided the precondition held — the implementer's obligation. An invariant is a condition that holds at every observable moment across an entire lifetime — you met loop invariants in CS 101 Lesson 3, and Lesson 3 of this course extends them to objects. CS 120 will write this trio as the Hoare triple {P} C {Q}; today we only need the division of labor.

2. Blame assignment (the widget)

The contract is a contract precisely because each failure mode has exactly one guilty party. Precondition false at the call site: the caller is wrong, no matter what happens next. Precondition true but postcondition false on return: the implementation is wrong. Run the combinations:

Notice the fourth combination: a precondition-violating caller meeting a buggy implementation. The contract renders no verdict on the implementation — outside the precondition, the function owes you nothing. Garbage in, anything out. That is not sloppiness; it is what makes the implementer's job finite.

Predict: withdraw(100, 250) returns -150 with the "correct" implementation. Is the implementation buggy?

No. The precondition (amount <= balance) was false, so the postcondition promises nothing — the implementation satisfied its contract vacuously. The bug lives at the call site. If you want the function itself to reject bad callers loudly, that is a different (and legitimate) contract: "raises ValueError if amount exceeds balance" — which turns misuse into a defined outcome. Lesson 6 is entirely about designing those outcomes.

3. Strong and weak specifications

Contracts trade off. Weakening a precondition (accepting more inputs) makes life easier for callers and harder for implementers; strengthening a postcondition (promising more) does the same. The classic trap is a postcondition that is accidentally weak. "Returns a sorted list" is satisfied by return []. The honest contract for sorting is a conjunction: the output is ordered and it is a permutation of the input. The second clause is the one everyone forgets, and the one that catches real bugs — you will demonstrate this to yourself in the explorations, and CS 350's property-based testing builds a career on it.

4. assert: contract checks, not error handling

def isqrt(n):
    assert n >= 0, "precondition: n must be non-negative"
    ...

Python's assert raises AssertionError when its condition is false — and is deleted entirely when the interpreter runs with -O. That single fact dictates its use: assertions document and spot-check contracts during development; they are never validation of untrusted input, because production flags can silence them. A user typing a negative number is not a contract violation — it is a normal, expected event that deserves designed error handling (Lesson 6). A colleague's code passing a negative number is a contract violation, and an assert that names the guilty clause is a gift to whoever debugs it.

5. Contracts are the unit of design

Everything in this course is contracts wearing different outfits. An abstract data type (Lesson 2) is a bundle of operation contracts. A representation invariant (Lesson 3) is a private contract between a class's methods. Substitutability (Lesson 4) is the rule that a subclass may not strengthen preconditions or weaken postconditions. A unit test (Lesson 7) is a contract clause made executable. When CS 101's final asked for an idempotent formatter, that was a postcondition: format(format(x)) == format(x). You have been signing contracts all along; now you get to draft them.
