Laziness — Computing Only What Is Consumed

Every collection you have returned from a function was a decision you may not have known you were making: compute everything, now, in memory. This lesson is about the other choice — values produced one at a time, on demand, only as far as the consumer pulls. Python's iterator protocol is the plumbing; generators are the ergonomic way to build it; and "lazy by default, materialize deliberately" is the design stance that falls out.

1. The protocol under every for loop

for item in things:        # is sugar for:
    ...                    #   it = iter(things)       # __iter__
                           #   while True:
                           #       try: item = next(it) # __next__
                           #       except StopIteration: break

An iterable can hand out iterators (__iter__); an iterator produces values (__next__) until it raises StopIteration. The distinction with teeth: a list is an iterable you can walk any number of times; an iterator is its own progress — a stream with no rewind. Every iterator is also iterable (returning itself), which is exactly why passing one around and looping twice silently does nothing the second time. The exploration makes you predict that ambush.

2. Generators: functions that pause

def countdown(n):
    while n > 0:
        yield n        # emit a value, FREEZE right here
        n -= 1

A def containing yield does not run when called — it instantly returns a generator object with the body frozen at the top. Each next() runs to the next yield and suspends: locals, position, the whole frame held in stasis. It is make_counter from Lesson 5 with the interpreter managing the state, or an iterator class with the boilerplate deleted. Two consequences worth scars: the body's validation code doesn't run at call time (your "fail fast" check fires at first next(), far from the bad call), and a return inside a generator means stop, not produce.

3. Pull, don't push (the widget)

Chain generators and you get a pipeline where the consumer sets the pace. Nothing runs until something pulls:

naturals = count(1)                        # 1, 2, 3, ... forever
squares  = (n * n for n in naturals)
odds     = (s for s in squares if s % 2 == 1)
# consumer: next(odds), five times

The infinite source is harmless: only what the consumer demands is ever produced. Note the pull cascading right to left — odds asks squares, squares asks naturals — and even squares being produced and immediately discarded by the filter. Constant memory, arbitrary stream.

4. Laziness as a design tool

The design payoff mirrors Lesson 5: generators separate production from consumption policy. A parser that yields records lets the caller decide whether to stream, count, or collect — without the parser growing limit= and as_list= parameters. Log processing, pagination, walking trees: same shape. itertools is the standard library's kit of lazy combinators (islice is lazy [:n], chain is lazy concatenation); skim its docs once so you know what not to reinvent. The final's parse_ledger is graded on being genuinely lazy — a probing test counts how far it reads.

Predict: numbers = iter([1, 2, 3, 4]); first = 3 in numbers. What does list(numbers) give afterward?

[4]. The membership test consumed the stream up to and including the 3 it found. The iterator has no memory of what passed through it and no way back. If you need to look twice, materialize: numbers = list(source) — that is a legitimate choice, made deliberately.

5. When to materialize

Laziness is a default, not a religion. Reach for list() when you need len, random access, multiple passes, or sorting (which must see everything anyway). And know laziness's honest cost: errors surface where the value is consumed, not where the pipeline was built, which moves tracebacks away from causes. Debugging a five-stage lazy pipeline is CS 220's pointer-chasing with nicer syntax — build pipelines short, name the stages, and materialize at boundaries you want to be able to inspect.
