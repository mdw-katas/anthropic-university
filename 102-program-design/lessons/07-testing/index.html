Tests Are Executable Specifications

You have practiced TDD for years; this lesson is the theory that makes the practice load-bearing. Lesson 1 said a contract has clauses; a unit test is one clause, compiled to something that can say no. That reframing settles most testing debates before they start: what to assert (the contract), what not to assert (the implementation), and why a test that can't fail is worse than no test at all.

1. Arrange, Act, Assert

class TestBag(unittest.TestCase):
    def test_removing_last_copy_removes_the_element(self):
        bag = Bag(["apple", "apple"])        # Arrange: a world
        bag.remove("apple", 2)               # Act:     ONE behavior
        self.assertNotIn("apple", bag)       # Assert:  the postcondition

One behavior per test, named as a sentence that states the clause. When this suite fails, the failure list reads as "which contract clauses are broken" — a specification with a pass/fail bit. The annihilate step (fresh fixtures per test; unittest builds a new instance per method) is what keeps tests independent: any order, any subset, same verdicts.

2. Test the contract, not the code

Lesson 2's abstraction barrier applies with full force: a good test suite exercises the public operations and would pass unchanged if you swapped the representation. The a1 test that checks "removing the last copy removes the element" survives your refactor from dict-of-counts to sorted list; a test asserting bag._counts == {...} dies — and worse, it fossilizes the representation, making the refactor expensive precisely because you tested it. Assert observable behavior; let the rep stay negotiable. (This is also why over-specified mock tests rot: they assert the conversation, not the outcome.)

3. Determinism is a design achievement (the widget)

A test that consults the real clock, real network, or real randomness imports the world's moods into your CI. The repair is never "retry the suite" — it is a seam: pass the dependency in, and pass a fake in tests. Run both suites a few times:

Simulated, but faithfully: the "real" suite asserts on timing that sometimes straddles a boundary. The fake clock makes the answer a fact, not a race. Determinism did not come from better assertions — it came from a better design: the clock became a parameter.

4. The doubles taxonomy

Double It does Use it to
Dummy Fills a parameter, never used Satisfy a signature
Stub Returns canned answers Feed the code inputs
Spy Records how it was called Assert on outputs/effects
Fake Real behavior, cheap implementation Replace clocks, filesystems, queues
Mock Pre-programmed expectations that self-verify Assert a protocol was followed

The names are Meszaros's (xUnit Test Patterns), and they name intents, not classes. Python's unittest.mock.Mock will happily play any of these roles — too happily: it answers every method call with a fresh truthy Mock, so a typo'd method or an assertion-free test passes in eerie silence. The exploration lets a mock "pass" a test that tests nothing. Prefer the dumbest double that can fail honestly; in a5 you will build them yourself — a spy, a stub, a fake clock — and then use your fake to test a retry loop without sleeping.

Predict: m = Mock(); if m.is_configured(): — which branch runs, given that is_configured was never set up?

The true branch, always. m.is_configured springs into existence on access, calling it returns another Mock, and every Mock is truthy. A test built this way cannot fail, and a test that cannot fail asserts nothing. The cure is boring: assert on concrete values, and make fakes that return real data.

5. Design for testability, get design for free

Code is testable when its collaborators arrive through the front door (parameters, constructors) instead of being conjured inside (time.time(), module globals, singletons) — the same dependency-direction discipline Lesson 9 preaches at module scale. The recipe with the best power-to-dogma ratio: a pure core (functions from values to values — trivially testable) wrapped in a thin imperative shell that does the I/O. CS 101's mdtable was secretly this shape: main() was free plumbing; every graded function was pure. The final repeats the trick at multi-module scale, and CS 350 will give the whole subject a semester.
