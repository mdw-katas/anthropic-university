Failure Is an Output — Design It

Half of production code is about things going wrong, yet "error handling" is usually whatever accumulated during incident reviews. This lesson treats failure as part of the contract: for every operation, the designer chooses which failures are normal outcomes, how they are reported — exception, sentinel, or result value — and who is responsible for recovery. Your sixteen years of Go say if err != nil; Python took the other fork in the road, and it pays to know both forks by name.

1. Exceptions: non-local, typed, impossible to ignore silently

An exception abandons the normal return path and unwinds the call stack frame by frame until a matching except catches it — or the program dies with a traceback. The three design-relevant properties: propagation is automatic (middle layers that can't help don't have to forward errors by hand), matching is by type (the hierarchy is part of your API), and ignoring one is loud (unlike a dropped error return, which is silent — the trade-off Go and Python made in opposite directions). EAFP — "easier to ask forgiveness than permission" — is the official house style: try the operation and handle the failure, rather than look before you leap, because between the look and the leap the world can change.

2. The machinery (the widget)

One statement, four clauses, and an exact protocol for who runs when. Step through each scenario:

Three rules worth engraving: except clauses are tried top to bottom, first match wins — so order subclasses before base classes or the base swallows everything. else runs only when no exception occurred, letting you keep the try block minimal (guard exactly the risky line, not the happy path after it). finally runs no matter what — return, raise, or success — which makes it the home of cleanup, and (see the exploration) a place where a stray return can silently eat an in-flight exception.

3. Exceptions have an abstraction level

class ConfigError(Exception): ...

def load_port(text):
    try:
        return int(text)
    except ValueError as err:
        raise ConfigError(f"port must be an integer, got {text!r}") from err

Lesson 2's barrier applies to failures too: a caller of load_port should catch ConfigError, not need to know that parsing happens with int(). raise ... from translates the exception at the boundary while chaining the original onto __cause__, so the traceback shows both stories. Catch precisely, translate at boundaries, and let everything you can't handle keep flying — a bare except: (which catches even KeyboardInterrupt) is how bugs get buried alive. Assignment a4 bans it by AST inspection, with prejudice.

4. Sentinels and result types: the other two forks

Strategy Example Right when
Exception "x".index("q") raises Failure is exceptional, or must not be ignorable
Sentinel "x".find("q") == -1 Absence is normal and handled at the call site
Result Ok(4250) / Err(reason) Failure is data — collected, mapped, reported in bulk

The standard library ships find/index as a matched pair on purpose: same question, two failure contracts. Sentinels bite when the sentinel is a legal answer (dict.get returning None for a key whose value is None — CS 101's old friend). Result types make failure a first-class value: a validator that checks fifty records shouldn't detonate on the third, it should return fifty results. Rust and Go made this the default; in a4 you will build a small Ok/Err and feel exactly what their compilers automate. CS 340 revisits this as the "error monad" — mention that at parties and see what happens.

Predict: a function returns "A" from its try block, but its finally block also says return "B". Which one does the caller get?

"B". finally runs after the try's return value is staged, and a return there replaces it — worse, if an exception was in flight, that return discards the exception entirely. Python 3.14 finally made this a SyntaxWarning. Keep finally for cleanup; never return from it.
