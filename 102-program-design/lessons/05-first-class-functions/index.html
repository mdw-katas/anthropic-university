Functions Are Values; Closures Are Objects Wearing Hats

You pass functions around constantly — handlers, callbacks, sort.Slice comparators. The precise version: in Python a function is an ordinary object, created by def at runtime, assignable, storable, returnable. And when a function is created inside another function, it closes over the enclosing variables — not their values, the variables themselves. That distinction is this lesson's entire booby trap, and its superpower.

1. Higher-order functions as design tools

sorted(courses, key=lambda c: c.credits)   # inject the POLICY
records.sort(key=str.lower)                # methods are values too

A higher-order function takes or returns functions. The design idea underneath: separate the mechanism (how to sort, how to walk, how to retry) from the policy (compare how? visit what? give up when?), and pass the policy in. map, filter, and reduce are the canonical trio — transform each, keep some, combine all — and CS 420 will show you the same trio running across a thousand machines under the name MapReduce. In Python, the comprehension is usually the idiomatic spelling of map+filter; the concept matters more than the builtin.

2. Closures: captured variables, not captured values

def make_counter():
    count = 0
    def increment():
        nonlocal count      # rebinding? announce it
        count += 1
        return count
    return increment

Each call to make_counter() mints a fresh count and a fresh increment holding it — private, persistent state with no class in sight. Two counters never interfere. Note nonlocal: CS 101 Lesson 4 taught that assignment makes a name local; nonlocal is how an inner function says "no — rebind the enclosing one." (Reading, or mutating a captured list, needs no declaration; only rebinding does.) A closure and an object are the same design atom — state plus behavior — differing mainly in how many behaviors share the state. One behavior: closure. Several: class.

3. The late-binding trap (the widget)

Because closures capture variables, a closure created in a loop sees the variable's value when called, not when created. Make the callbacks, then click them:

buggy: capture the loop variable

callbacks = [(lambda: i) for i in range(3)]

fixed: freeze the value with a default argument

callbacks = [(lambda i=i: i) for i in range(3)]

All three buggy callbacks share the one variable i, which finished the loop at 2. The default argument is evaluated at definition time, giving each lambda its own frozen copy — the standard idiom for this repair.

Predict: def append_to(item, bucket=[]) — call it twice as append_to(1) then append_to(2). What does the second call return?

[1, 2]. Same mechanism as the fix above, other edge of the blade: the default is evaluated once, at definition — so every default-using call shares one list. Mutable defaults are the most-reported Python gotcha in existence; the idiom is bucket=None plus if bucket is None: bucket = [].

4. Decorators are just this, wearing syntax

def logged(fn):
    def wrapper(*args, **kwargs):
        print(f"calling {fn.__name__}")
        return fn(*args, **kwargs)
    return wrapper

@logged            # sugar for: fetch = logged(fetch)
def fetch(url): ...

A decorator is a higher-order function applied with @. No new semantics — wrapper is a closure over fn. Memoization, retry, timing, auth checks: all "wrap the mechanism, keep the signature." Assignment a3 has you build the classics — compose, memoize, once — from bare closures, with the standard library's versions banned by AST inspection, because you should build each of these once in your life before being allowed to import them forever after.
