The Shape of a Program Is Its Dependency Graph

Zoom out far enough and every design idea in this course becomes one picture: modules as nodes, imports as arrows. Cohesion is what belongs inside a node; coupling is how many arrows exist; architecture is which way they point. You have felt good and bad shapes for years — this lesson gives the feelings coordinates, just in time for the final project, which is a four-module program whose shape is part of the grade you assign yourself.

1. A module is an ADT at file scale

Everything Lesson 2 said about classes applies to files. A module's interface is the set of names it means to export; its implementation is everything else, flagged by the same _underscore convention (which from module import * respects, and honorable colleagues do too). A module with a describable purpose — "owns the Transaction model," "turns text into Transactions" — is cohesive. A module named utils is a confession that no one decided.

2. Coupling: count what a change touches

Two modules are coupled to the degree that changing one forces changes in the other. Import contracts (call functions, catch documented exceptions) and coupling stays loose. Reach for another module's underscored internals, or share mutable module-level state, and you have built one big module with extra filenames. The test from Lesson 7 doubles as a coupling meter: if you cannot test parsing without constructing a report, the arrow between them is load-bearing in a way the design never intended.

3. Dependency direction (the widget)

Arrows are not neutral. Dependencies should point from the volatile toward the stable — from I/O and formatting decisions toward the domain model, never back. Here are the final project's four modules. Click arrows on and off; the analyzer reports layers and cycles:

Try adding model -> report. The analyzer objects — and so does Python: a genuine import cycle detonates at startup with a partially initialized module. The deeper objection is conceptual: if the MODEL knows how it is FORMATTED, every report tweak risks the domain. Stable things must not depend on volatile things.

4. The intended shape, read aloud

cli  ->  parsing  ->  model
cli  ->  report   ->  model

model imports nothing and knows nothing — pure data and invariants, the still center. parsing knows how text becomes model objects; report knows how model objects become text for humans; neither knows the other exists. cli, the most volatile and least interesting module, knows everyone — it is the imperative shell from Lesson 7, wiring streams to functions. Count what a change touches: new report format — one module; new syntax — one module; new field on Transaction — the model plus whoever consumes the field, flagged loudly. That accounting exercise is architecture; CS 350 scales it to teams, and CS 340 will show you languages that enforce the arrows.

Predict: module a.py has `import b` at its top, and b.py has `import a`. What actually happens when you run `python a.py`?

Usually an AttributeError (or ImportError) from a partially initialized module. Running a starts executing it; at import b, control enters b; at b's import a, Python sees a already in sys.modules (CS 101 Lesson 7's import-once rule) and hands back the half-built module object — anything a defines below its import statement doesn't exist yet. Whether it crashes depends on timing and luck, which is the worst possible answer. Cycles are not a style violation; they are a live grenade.

5. The case study starts now

The final project, ledger, is the whole course in one program: a plain-text accounting tool with a model module built on representation invariants (Lessons 1–3), a parser with a designed exception hierarchy (Lesson 6) that yields transactions lazily (Lesson 8), a report module of pure functions (Lessons 5, 7), and a free cli.py already wired in the shape above (this lesson). The test suite checks behavior and an algebraic invariant that double-entry bookkeeping shares with physics: everything conserves. Read the README before writing any code; design the model first.
