Abstract Data Types — Program to the Promise, Not the Parts

Every time you use a Python dict, you rely on operations whose contracts you know cold and an implementation (open-addressed hash table, as of this writing) you may never have read. That arrangement — a type defined entirely by its operations and their contracts, with the representation hidden behind them — is an abstract data type. Barbara Liskov formalized it in 1974 building the CLU language, and it is the single most load-bearing idea in this course.

1. An ADT is a bundle of contracts

A stack is not "a list you use from one end." A stack is: whatever type satisfies these operation contracts —

Operation Contract
push(x) Afterward, peek() returns x; size grows by 1
pop() Removes and returns the most recent unpopped push; size shrinks by 1; requires nonempty
peek() Returns what pop() would, without removing it; requires nonempty
len(s) Number of pushed-but-not-popped items

Nothing in that table mentions lists, arrays, or pointers. Lesson 1 gave you contracts for functions; an ADT is the same discipline applied to a whole type. The operations are the entire public story — which is why the set of operations you choose is the design decision.

3. Duck typing is Python's interface mechanism

You know this move from Go: a type satisfies an interface by having the methods, not by declaring allegiance. Python goes further — the check happens per call, at runtime. Any object with push, pop, peek, and __len__ is a stack as far as client code is concerned. There is no compiler to enforce the contract's semantics, though; a pop that returns the oldest item has the right shape and the wrong soul. (CS 340 studies type systems that can promise more; Lesson 7 shows how tests pin down the semantics that types cannot.)

Predict: drain pops until empty. It works on a list and on our stacks. What happens on a dict — say, drain({"a": 1, "b": 2})?

dict has a pop method and a length — the duck looks right — but dict.pop requires a key argument, so the zero-argument call raises TypeError at the moment of the call. Duck typing checks nothing in advance; every method call is its own audition. The exploration makes you predict this before running it.

4. Rep exposure: handing clients a crowbar

class Roster:
    def __init__(self):
        self._names = []
    def members(self):
        return self._names          # <- the crowbar
        # return list(self._names)  # <- the fix: defensive copy

Return your internal mutable state directly and every caller holds a reference that bypasses all your methods — your invariants can be broken by code you have never met, in a stack trace you will not enjoy. The defensive copy costs a little; debugging spooky action at a distance costs an afternoon. This is CS 101 Lesson 6's aliasing lesson, promoted from accident to design concern. Assignment a1 will grade you on it.

5. Why the barrier pays rent

The abstraction barrier is what makes change survivable. The list-backed stack is fine until profiling says otherwise; swap in something cleverer and every honest client keeps working, proven by the operation contracts alone. CS 201 spends a semester exploiting exactly this: same interface, radically different cost profiles. Design the promise once; renegotiate the parts whenever you like.
