Inheritance Is a Contract, Composition Is a Part

Object-oriented design has one genuinely magic trick — dynamic dispatch — and two ways to reuse code, one of which is oversold. This lesson names the trick precisely, shows exactly how Python performs it, and gives you principled reasons (not slogans) for the advice you have been nodding along to for years: favor composition over inheritance.

1. Dynamic dispatch, precisely

When Python executes obj.describe(), the method that runs is chosen at call time by the runtime type of obj — not by anything visible at the call site. The call site names a message; the receiver supplies the meaning. That late choice is polymorphism's whole mechanism, and it is the reason old code can call new code: a function written last year, dispatching on describe(), runs a class you write today. (Go does this only through interface values; Python does it on every attribute access.)

2. Method lookup: the MRO (the widget)

Inheritance adds a search path. Python linearizes the class hierarchy into a Method Resolution Order and walks it until a class supplies the method. Pick an instance and a method; watch the walk:

class Animal:            speak, name
class Dog(Animal):       speak
class Puppy(Dog):        name
class Robot:             speak
class RoboDog(Dog, Robot):   (defines nothing)

RoboDog's MRO is RoboDog, Dog, Animal, Robot — bases are searched depth-first-ish, left to right, each class exactly once (C3 linearization). Note Robot.speak never wins: Dog got there first. super() means "the next class in the MRO," not "my parent" — which is why cooperative super().__init__() chains work even in diamonds.

3. When inheritance is the wrong tool

Substitutability is the real rule. Sub may inherit from Base only if every place that works with a Base works — contract intact — with a Sub. In Lesson 1's vocabulary: an override may weaken preconditions or strengthen postconditions, never the reverse. This is the Liskov Substitution Principle, and the canonical corpse is Square(Rectangle): mathematically a square is-a rectangle, but a rectangle's contract says width and height vary independently, and Square cannot honor it. The exploration lets you watch a perfectly innocent function get wrong answers from a Square it was promised was a Rectangle.

The fragile base class. Inheriting an implementation couples you to its private choices. Subclass dict, override __setitem__, and discover that __init__ and update never call your override — CPython's builtins call each other internally without dispatching through subclass methods. Your "logging dict" logs some writes. The exploration makes you predict how many.

4. Composition: the boring technique that keeps working

class LoggingDict:
    """HAS a dict; IS not a dict."""
    def __init__(self):
        self._store = {}
        self.log = []
    def __setitem__(self, key, value):
        self.log.append(key)
        self._store[key] = value
    def __getitem__(self, key):
        return self._store[key]

The wrapper cannot be bypassed, because the inner dict is reachable only through it — the fence from Lesson 3 again. You couple to the dict's public contract instead of its internals, and you expose only the operations you actually support. The cost is delegation boilerplate; the benefit is that nothing you didn't write can surprise you. Rule of thumb worth its cliché status: inherit an interface (an abstract "is-usable-as"), compose an implementation (a concrete "is-built-from").

Predict: LoggingDict(dict) overrides __setitem__ to log. Then LoggingDict(a=1) followed by d["b"] = 2 followed by d.update(c=3). How many writes were logged?

One. Only the explicit d["b"] = 2 travels through your override; the constructor and update write through internal C paths. If you need a mapping that reliably intercepts writes, either compose, or subclass collections.UserDict — a class that exists precisely because of this trap.

5. Polymorphism without hierarchy

Here is the punchline for assignment a2: dispatch does not require inheritance at all. Any object with the right methods participates — Lesson 2's duck typing — so a "hierarchy" in Python is often just a set of unrelated classes agreeing on a protocol. Code that type-switches (if isinstance(node, Add): ...) forfeits the magic: it must be edited for every new type. Code that dispatches (node.evaluate(env)) is open to extension and closed to modification — new types plug into old code with zero edits. a2's test suite enforces this by injecting a node type you have never seen into your expression tree. Design accordingly.
