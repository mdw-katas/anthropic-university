Informal Efficiency: Counting Steps

The question that separates engineering from hoping: what happens to the cost when the input grows? This lesson builds the instinct informally; CS 201 and CS 301 will formalize it as big-O. The instinct is 90% of the daily value.

1. Count steps, not seconds

Seconds depend on the machine, the load, the phase of the moon. Steps — how many times the loop body runs — depend only on the algorithm and the input size n. So we count steps, keep only the dominant term, and drop constant factors: 3n² + 10n + 7 is "about n²", because for large n the n² term is the whole story.

3. Where steps hide

The loop you wrote is not the only loop that runs. Two famous ambushes:

if name in huge_list:        # a LOOP in disguise: scans the whole list, ~n steps
if name in huge_set:         # hash lookup: ~constant. Same syntax!

total = ""
for part in parts:           # looks linear...
    total += part            # ...but rebuilds the string each time: ~n² total
"".join(parts)               # linear. The idiom exists for a reason.

The instinct to develop: for every innocent-looking operation, ask "and what does that cost?" — especially membership tests, string concatenation, and anything inside a loop that touches a collection.

Predict: your a1 binary search on a million sorted items — about how many elements does it examine?

About 20 (2²⁰ ≈ 1,048,576). That's the probe test's secret: it allowed 60 accesses on 10,000 items, generous headroom over the ~14 a real binary search needs, and hopeless for a 10,000-step scan.

4. When n² is fine

Honesty clause: for n = 50, n² is 2,500 steps — nothing. Clarity beats cleverness until n is large or the code is hot, and measured beats assumed always (profilers arrive in CS 220). The sin is not writing a quadratic loop; it is not knowing you did.
