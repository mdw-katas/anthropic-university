What Is Computation?

You have shipped software for sixteen years, so you have done computation daily. This lesson asks the question you were never forced to answer precisely: what is the thing you were doing?

1. Algorithms, precisely

An algorithm is a finite, unambiguous, effective procedure that transforms input into output. Each word is load-bearing:

  • Finite — it terminates. A procedure that may run forever is not (yet) an algorithm for the problem.
  • Unambiguous — each step has exactly one meaning; no step requires judgment or taste.
  • Effective — each step is mechanically executable: a person with pencil and paper could do it, however slowly.

Notice what is not in the definition: computers. Algorithms predate them by millennia. The definition's most famous citizen is more than 2,300 years old.

2. A worked classic: Euclid's algorithm

To find the greatest common divisor of two positive integers: repeatedly replace the larger with the remainder of dividing larger by smaller, until one of them is zero. The other is the GCD.

def gcd(a, b):
    while b != 0:
        a, b = b, a % b
    return a

Step through it below. Watch the pair (a, b) — that pair is the algorithm's entire state, an idea we define next.

stepaba mod b

Predict before you peek: how many steps for gcd(8, 13)?

Six: (8,13) → (13,8) → (8,5) → (5,3) → (3,2) → (2,1) → (1,0). Consecutive Fibonacci numbers are Euclid's worst case — a fact that, in CS 301, becomes a proof that the algorithm runs in logarithmic time.

3. State: the noun your program is

The state of a running program is everything it would take to freeze it and later resume it perfectly: the values of all variables plus the position in the code. In the stepper above, the whole state fits in one table row. Every bug you have ever fixed was a discrepancy between the state you believed and the state that was.

A state machine is the simplest model of computation: a finite set of states, and rules for moving between them on each input. A traffic light. A regex engine. A TCP connection. CS 370 will prove exactly what this humble model can and cannot compute.

4. The stored-program idea

The founding insight of modern computing (von Neumann, 1945, building on Turing, 1936): a program is just data in the same memory as any other data. Numbers that encode instructions. The consequences are the industry you work in:

  • Programs can read programs — so compilers, linters, and formatters can exist.
  • Programs can write programs — so code generators and, lately, large language models can exist.
  • Programs can be wrong data — so buffer-overflow exploits can exist.

5. A first taste of cost

Two algorithms can be equally correct and unequally usable. Count the steps, not the seconds: checking every pair in a list of n items costs about n² steps; scanning it once costs n; Euclid's remainders shrink so fast the cost is closer to log n. This instinct — asking how does cost grow as input grows? — is the discipline's sharpest tool, formalized in CS 201/301.
