Recursion

A recursive function solves a problem by calling itself on a smaller version of the same problem. That single idea underlies parsers, file-system walkers, sorting algorithms, and the tree-shaped half of computer science. It is also just Lesson 3's termination argument and MATH 120's induction, wearing a third costume.

1. The two obligations

def factorial(n):
    if n == 0:            # 1. BASE CASE: an answer with no self-call
        return 1
    return n * factorial(n - 1)   # 2. PROGRESS: strictly smaller input

Every correct recursion discharges both: a base case that answers directly, and recursive calls that make measurable progress toward it. Miss either and you get Python's RecursionError — the interpreter's guard rail (default: ~1000 frames) against unbounded stack growth, each frame being exactly the stack frame from Lesson 4.

2. Trust the contract

The mental shift that makes recursion easy: when writing factorial(n), do not trace into factorial(n - 1). Assume it works — it satisfies its contract — and ask only: given a correct answer for n-1, is my answer for n right? That assumption is not faith; it is the induction hypothesis, and MATH 120 will let you prove it legitimate. Tracing into recursive calls is how recursion stays hard; contract thinking is how it becomes routine.

3. The shape of the calls

One self-call makes a chain (factorial). Two make a tree — and trees can explode. Naive Fibonacci recomputes the same subproblems exponentially many times. Expand the call tree yourself; every duplicated subtree is wasted work:

fib(2) appears three times in fib(5)'s tree. Dynamic programming — CS 301 — is the art of computing each subproblem once. One dict of remembered answers turns 15 calls into 6.

Predict: how deep does factorial(5)'s stack get, counting factorial frames only?

Six frames at the deepest moment: factorial(5) through factorial(0), all live simultaneously while factorial(0) computes. Depth is the cost recursion pays for its elegance — a chain of n calls needs O(n) stack.

4. Recursion ⇄ iteration

Anything iterative can be written recursively and vice versa — the two are formally equivalent (CS 370 territory). Practically: Python does not optimize tail calls, so linear chains (factorial, sum-of-list) are usually better as loops; tree-shaped problems (nested data, divide and conquer, file trees) are where recursion is the honest notation and iteration requires managing an explicit stack by hand — which is assignment a5's punchline, one lesson early.
