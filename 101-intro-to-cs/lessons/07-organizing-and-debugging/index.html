Modules, Testing, and Debugging as Science

Programs grow. This lesson is about the three habits that let them grow without rotting: organizing code into modules, pinning behavior with tests, and debugging by experiment instead of by vibes. You practice all three professionally; here we name the underlying rules precisely.

1. Modules: files are namespaces

Every .py file is a module — a namespace of its own (the "G" in LEGB). import counter does three things: finds the file, runs its top-level code, and binds the name. Two rules with consequences:

  • Import runs the module — once. Python caches modules in sys.modules; a second import anywhere in the process is a cheap cache hit, and top-level code does not run again. Module top-level is therefore for definitions, not for work.
  • __name__ tells a module how it's being used: "__main__" when run as a script, its own name when imported. Hence the idiom guarding every script's entry point — it makes the file importable (testable!) without side effects:
if __name__ == "__main__":
    main()

2. Testing: the contract, executed

You've now read a dozen unittest suites in this course; here is the anatomy, named. Each test is Arrange, Act, Assert — set up state, do the one thing, check the one outcome. Good tests are independent (order never matters), deterministic (no clocks, no randomness, no network), and specific (a failure points at one contract clause). The course grader is nothing but unittest discover plus arithmetic — infrastructure you could now build yourself.

class TestGCD(unittest.TestCase):
    def test_zero_conventions(self):
        self.assertEqual(algorithms.gcd(7, 0), 7)   # arrange+act+assert in one line

3. Reading a traceback like a local

A traceback is the call stack from Lesson 4, printed at the moment of death. Read it bottom-up: the last line is what went wrong; the frame above it is where; everything higher is how execution got there. Click each line:

Traceback (most recent call last):
File "report.py", line 12, in main
table = summarize(rows)
File "report.py", line 7, in summarize
return total / len(rows)
ZeroDivisionError: division by zero
Click any line of the traceback.

4. Debugging is hypothesis testing

Debugging by vibes: change things until it works. Debugging as science:

  1. Observe — reproduce the failure, minimally. A bug you can't reproduce on demand isn't understood yet.
  2. Hypothesize — one specific, falsifiable claim: "rows is empty because the filter drops everything when the date is a weekend."
  3. Experiment — the cheapest observation that could prove the claim false: a print, an assert, one debugger breakpoint, a unit test with a weekend date.
  4. Conclude and repeat — confirmed? Fix it and add the test that would have caught it. Refuted? You just learned something true; form the next hypothesis.

The discipline scales down (one print statement) and up (bisecting ten thousand commits). What it forbids is the shotgun: changing code you don't have a hypothesis about.

Predict: module config.py prints "loading!" at top level. Scripts a.py and b.py both import it, and a.py also imports b.py. Run a.py — how many "loading!" lines?

One. First import runs the module; every later import in the same process hits the sys.modules cache. (Run b.py directly instead and you'd also see one — it's per-process, not per-file.)
